Combat System — Hitboxes, State Machine & Anti-Cheat

A fully server-authoritative M1 combo system with critical attacks, blocking, parrying, and endlag. The client handles animation and input only — all damage, validation, and state lives on the server.

Timeline
2024 — Ongoing
Language
Lua / Luau

The Problem

Client-side combat is exploitable by design

In most beginner Roblox games, combat is handled entirely on the client — the player's machine decides when a hit lands, how much damage is dealt, and whether the attack was valid. This is easy to exploit. Any player can fire a remote event with fake hit data and deal unlimited damage.

The challenge was building a combat system where the client never has authority over damage or state — it only drives visuals and input — while still feeling responsive and snappy to play.

Core constraint: The server must validate every hit, manage every state transition, and be the single source of truth — but the player must never feel lag or desync between what they see and what the server knows.
Architecture

Clean client/server split

The system is split across six modules with clearly defined responsibilities. No module crosses its boundary — the client never touches damage, the server never touches animation.

InputHandler
Reads keys
Plays animations
Fires RemoteEvent
→
CombatServer
Validates hits
Applies damage
Manages combo state
→
StateController
Tracks Attacking
Blocking, Parrying
Endlag, Critical

InputHandler (client) reads player input, plays the animation immediately for responsiveness, then waits for the hit timestamp before casting the hitbox and firing to the server. This means the server receives the event at the exact frame the hit would land — not before.

CombatServer receives the event, runs distance validation on every reported humanoid, checks the player's current state (can't attack while blocking, in endlag, etc.), applies damage, and fires a StateUpdate back to all relevant clients.

StateController is a lightweight key-value store per entity. States like Attacking, Endlag, and Parrying can be set with optional auto-expiry durations so they clean themselves up automatically.

Key Decisions

Why deferred hitbox casting?

A naive implementation fires the hitbox the moment the player clicks. But animations have a wind-up — the sword hasn't reached the target yet. Casting immediately means hits register before the visual swing, which feels wrong and is easy to abuse.

Each hit in the style data carries a hitTimestamp — the number of seconds after the click when the hit frame fires. The client defers the hitbox cast using task.delay(hitTimestamp), so the spatial query runs at the exact moment the animation connects visually. 

-- Defer hitbox cast + server fire to the actual hit frame
pendingHitboxTask = task.delay(hitData.hitTimestamp or 0, function()
    pendingHitboxTask = nil
    local char = player.Character
    local hits = char
        and HitboxModule.cast(char, hitData.hitboxRadius, hitData.hitboxOffset, 0.15)
        or {}
    combatRemote:FireServer({ type = "Combat", action = "M1", hits = hits })
end)

Server-side distance validation

Even though the client casts the hitbox, the server never trusts those results blindly. Every humanoid in the reported hit list is checked against the attacker's position server-side. If the distance exceeds MAX_HIT_DISTANCE (10 studs), the hit is discarded — no damage, no feedback. 

local function validateHits(attacker, hits)
    local confirmed = {}
    for _, humanoid in ipairs(hits) do
        local dist = (attackerHRP.Position - targetHRP.Position).Magnitude
        if dist <= MAX_HIT_DISTANCE then
            table.insert(confirmed, humanoid)
        end
    end
    return confirmed
end

Blocking vs Parrying — timing matters

When a player holds the block key, they enter a Parrying state for 150ms before transitioning to a sustained Blocking state. A hit that lands during the parry window deals zero damage and triggers a special parry animation. A hit during regular blocking deals 40% reduced damage. A token system cancels the parry-to-block transition if the player releases the key early.

Anti-Cheat

Rate limiting all combat events

Every combat remote event passes through a RateLimiter before any logic runs. Each player gets a rolling window — max 30 actions per 5 seconds. Timestamps outside the window are pruned on each check. Players who exceed the limit are warned and their action is silently dropped.

This doesn't just catch speed hackers — it also prevents edge cases where client bugs could spam the server with repeated events.

Design Pattern

Style data as configuration, not logic

Each combat style (sword, fist, etc.) is defined as a pure data module — no logic, just configuration. Combo length, hit damage, hitbox sizes, animation names, and flourish functions are all declared in one place. The server loads the correct style for each player dynamically via PlayerData.GetStyleData().

This means adding a new weapon style requires zero changes to the combat server — you just write a new data module and drop it in the right folder. 

-- Base style data (data only, no logic)
hits = {
    [1] = { damage = 7,  animName = "Base_M1_1", hitboxRadius = 2.5, hitTimestamp = 0.2 },
    [2] = { damage = 7,  animName = "Base_M1_2", hitboxRadius = 2.5, hitTimestamp = 0.2 },
    [3] = { damage = 7,  animName = "Base_M1_3", hitboxRadius = 2.5, hitTimestamp = 0.2 },
    [4] = { damage = 10, animName = "Base_M1_4", hitboxRadius = 3,   hitTimestamp = 0.25,
            flourish = knockback },  -- last hit triggers knockback
},
Reflection

What I learned

Building this made the tradeoffs of client-server architecture very concrete. Every time I gave the client more authority, I introduced an exploit vector. Every time I moved logic to the server, I had to think carefully about latency and how to keep the experience feeling responsive.

The deferred hitbox pattern was the most important insight — responsiveness and server authority aren't mutually exclusive if you time your events correctly. The client plays the animation instantly; the server just receives the event a fraction of a second later at the right moment.
