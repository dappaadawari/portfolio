Inventory & Datastore System — Session Locking & Data Safety

Persistent player data with session locking, versioned saves, backup keys, and autosave. A fully reactive inventory UI with stackable items, weight limits, and server-validated world item pickups.

The Problem

Player data is the most fragile part of any game

DataStore failures, server crashes, and players joining from multiple servers simultaneously can all corrupt or overwrite player data. Most beginner implementations just call SetAsync on PlayerRemoving and hope for the best — no locking, no retries, no backups.

The goal was to build a data layer that could survive edge cases: a player rejoining immediately after a crash, a DataStore outage mid-session, or two servers racing to save the same player's data.

Core constraint: Data must never be lost or duplicated, even if the server shuts down unexpectedly, the DataStore is temporarily unavailable, or a player joins two servers at once.
Session Locking

Preventing multi-server data conflicts

When a player joins, DataManager acquires a session lock using UpdateAsync on a separate lock store. The lock stores the server's JobId and a timestamp. If another server already holds a fresh lock for that player, the joining player is kicked with a message to rejoin shortly.

Locks older than 300 seconds are treated as stale (e.g. from a crashed server) and overwritten automatically. On PlayerRemoving, the lock is released before the session data is wiped.

return lockStore:UpdateAsync("Lock_" .. userId, function(existing)
    if existing == nil then return { jobId = jobId, timestamp = now } end
    if existing.jobId == jobId then return { jobId = jobId, timestamp = now } end
    if (now - (existing.timestamp or 0)) > LOCK_EXPIRY then
        return { jobId = jobId, timestamp = now } -- stale, overwrite
    end
    return nil -- active lock on another server, abort
end)
Data Safety

Backups, retries, and versioning

Every save writes a backup of the previous save to a separate backup DataStore before overwriting the main key. This means a failed save never destroys the last good state — the backup always reflects the last confirmed write.

All DataStore calls are wrapped in a retry loop — up to 3 attempts with 2-second delays. If all retries fail on load, the player is kicked rather than allowed to play on empty data. On save failure, a warning is logged but the session continues.

Save data carries a version number. When a player loads, if their saved version is older than the current schema, a migration chain runs — each step upgrading one version at a time — before the data is validated and populated into the session.

Inventory Design

Stackable items, weight, and world drops

The inventory supports both stackable and non-stackable items. Adding a stackable item searches for an existing slot with room before creating a new one. Each item has a weight value — the player's total carried weight is tracked and capped, with weight attributes synced to the character for the client to read reactively.

When a player drops an item, WorldItemManager spawns a plain Model in the workspace (not a Tool, to avoid Roblox's auto-pickup) with a ClickDetector. On click, the server does a radius check — if the player is within 10 studs, the item is claimed atomically (removed from tracking before AddItem is called) to prevent two players picking it up in the same frame.

Client UI

Reactive inventory with race condition guards

The inventory client fetches a snapshot of the player's inventory on load via RemoteFunction, then listens for AddItem and RemoveItem RemoteEvents. Events that arrive before the initial snapshot finishes loading are queued and replayed in order once setup is complete — preventing duplicate nodes or missed updates during the load window.

Each inventory node has a duplicate guard: if a node for a given UID already exists, it updates the amount label rather than spawning a second GuiObject. Hover tooltips show item name, description, and rarity, and are cleaned up immediately when the last item is removed.
