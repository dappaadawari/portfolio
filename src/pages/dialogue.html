Dialogue System: Node-Based Branching NPC Conversations

A reusable branching dialogue system driven by node graphs. Any NPC can use it by swapping a module. Supports dynamic text injection, scrollable response buttons, and clean server/client separation.

Timeline
November 2025
Language
Lua / Luau
Platform
Roblox Studio
Type

The Problem

Designing a main driver that can handle dialogue for any npc, and support brancing and looping

A common beginner mistake is building dialogue directly into each NPC's script, unique UI handling, unique response logic, unique branching and event handling. To me, this doesn't scale the way I want it, and handling 50+ npcs like this is inconvinient. The goal was a single system that any NPC in the game could use just by pointing at a data module.

Design goal: Adding a new NPC with full branching dialogue should require writing only a data module, zero changes to the core dialogue system itself.
Node Graph Structure

Conversations as data

Each NPC's dialogue is defined in a module script as a flat table of named nodes. Every node has a text field and a responses array. Each response has an id, display text, and a next field pointing to the next node name or nil to end the conversation.

Nodes = {
    ["Start"] = {
        text = "Hey there, stranger. What's your deal?",
        responses = {
            { id = "AskWho", text = "Who are YOU?",        next = "Who_Are_You" },
            { id = "Intro",  text = "I'm {playerName}.",    next = "Player_Intro" },
            { id = "End",    text = "Uh... bye.",          next = nil }
        }
    },
}

The server caches all dialogue modules on startup. When a player interacts with an NPC, the server looks up the module by name, sends the Start node to the client, and from that point the client drives the UI while the server advances the graph on each response.

Dynamic Text

Player name injection at display time

Dialogue text can contain template tags like {playerName} or {displayName}. Before rendering any text the client runs it through a replacement function, swapping tags for the local player's actual values. This means NPC dialogue can feel personal without storing any player-specific data in the modules. and support greater things like events in more advanced versions.

Reusability

Adding a new NPC takes minutes

To add a new NPC to the game you create one module script following the node structure, name it NPCName_Dialogue, and place it as a child of the DialogueController. No other code needs to change. The system discovers and caches all dialogue modules automatically on startup.

The same UI, the same server logic, and the same RemoteEvent handle every NPC in the game. This was the most important architectural decision, keeping data and behavior fully separate.
